## 1. 文档目的与结论摘要

### 1.1 目的

在 [立项交接](./ZestLLM-立项交接.md) 基础上，结合市面成熟产品、可插拔 SPI 架构、**Admin + Admin-UI**、**Spring Boot + MyBatis-Plus + 阿里规范**、**zestflow-mcp 元件式开发**，评估 ZestLLM **能否做、怎么做、风险在哪**。

### 1.2 总体结论

| 维度 | 结论 | 置信度 |
|------|------|--------|
| **整体可行** | ✅ 推荐立项，采用组装式架构 | 高 |
| **技术栈** | ✅ 与 ZestFlow / Zestory 已验证栈一致 | 高 |
| **开源底座** | ✅ LiteLLM + Langfuse + MySQL + Valkey 组合成熟 | 高 |
| **可插拔 SPI** | ✅ Spring Boot 条件装配模式成熟，增量成本可控 | 高 |
| **Admin-UI** | ✅ 复用 zestflow-admin-ui 工程模式，4~6 周可出 MVP 页 | 中高 |
| **MCP 元件开发** | ✅ zestflow-mcp 0.1.0 已发布，zestory 已实战 | 高 |
| **性能目标** | ✅ 三段式 + 不经 CP 转发 token 可达 KPI | 中高 |
| **主要风险** | 输出 Schema 稳定性、LiteLLM 上游变更、CP/Admin 职责边界 | 中 |

**一句话建议：** 以 **zest-llm-admin（Admin + Runtime 合一）+ SPI 可插拔 + zestflow-mcp 驱动开发** 为 v0.1 路径，**不自研网关与观测**，4~5 周出可演示 MVP。

**结论：** 除 LiteLLM/Langfuse 需集成验证外，主栈均在团队内跑通过，**技术可行性高**。

### 3.2 关键集成验证项（PoC，建议 3~5 天）

| # | PoC 项 | 成功标准 | 优先级 |
|---|--------|----------|--------|
| P1 | LiteLLM `chat/completions` 调用 DeepSeek/OpenAI | 返回 content + usage | P0 |
| P2 | `@ZestLLM` AOP 调 CP invoke 映射 Result | Demo 单测通过 | P0 |
| P3 | Admin JWT + Runtime appToken 双鉴权 | 越权测试失败 | P0 |
| P4 | Flyway 8 表初始化 + MyBatis CRUD | 集成测试通过 | P0 |
| P5 | admin-ui build 嵌入 admin static | 浏览器可访问 | P1 |
| P6 | Langfuse trace 上报 | Langfuse UI 可见 span | P1 |
| P7 | SPI 切换 `observability: noop` | 无 Langfuse 仍可 invoke | P1 |
| P8 | zestflow-mcp init-dev + scaffold_component | 生成合法 Handler | P1 |

PoC 全部通过即可进入 v0.1 正式开发。

### 3.3 与 Spring AI 的关系

| 方案 | 评估 |
|------|------|
| 完全依赖 Spring AI | ❌ 缺 Control Plane 与 Admin，且 API 迭代快 |
| 完全不用 Spring AI | ✅ 默认方案，LiteLLM HTTP 即可 |
| Spring AI 作为可选 SPI | ✅ 预留 `ModelGatewayAdapter` 实现，v0.3+ |

**结论：** Spring AI 作 **可选 SPI**，不作为核心依赖。

## 4. 成熟开源底座评估

### 4.1 LiteLLM Proxy（默认 ModelGateway）

| 项 | 评估 |
|----|------|
| 许可证 | MIT ✅ |
| 能力 | 100+ 模型、OpenAI 兼容 API、fallback、RPM/TPM 限流 |
| 运维 | Docker 官方镜像，配置 YAML |
| 风险 | 上游 API 变更、大版本升级需回归 |
| 替代 | OneAPI（MIT）、直连 OpenAI SDK |
| 结论 | **✅ 作为默认网关可行** |

### 4.2 Langfuse（默认 Observability）

| 项 | 评估 |
|----|------|
| 许可证 | MIT ✅ |
| 能力 | Trace、成本、Prompt 实验、评估数据集 |
| 运维 | self-host 需 PG + ClickHouse，资源高于 LiteLLM |
| 风险 | 部署复杂；v0.1 可用 `noop` 跳过 |
| 替代 | OpenTelemetry + Jaeger、纯 DB 审计 |
| 结论 | **✅ v0.2 接入；v0.1 可 Noop** |

### 4.3 MySQL + Valkey

| 项 | 评估 |
|----|------|
| 用途 | MySQL 存元数据与 Execution；Valkey 缓存 RoutePolicy |
| 与 ZestFlow 差异 | 与 ZestFlow 统一 MySQL 栈，独立 `zest_llm` 库，避免耦合 |
| 结论 | **✅ 可行**（禁止 H2 / PostgreSQL） |

### 4.4 底座组合对比

| 组合 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| LiteLLM + Langfuse + PG + Valkey | 能力完整、许可证友好 | 组件多 | **生产推荐** |
| LiteLLM + Noop + PG + Caffeine | 部署最简 | 无 Trace UI | **MVP 推荐** |
| OneAPI + 自研日志 | 国内渠道熟悉 | 缺 Prompt Lab | 备选 |

## 5. 可插拔 SPI 可行性

### 5.1 为什么可行

1. **Spring Boot 条件装配** 是成熟模式（ZestFlow 已有 `EventCollector` SPI、`RouteStrategy` SPI）
2. **接口粒度合理**：网关、观测、缓存、配额各自独立，替换不影响领域服务
3. **测试友好**：Noop / InMemory 实现可覆盖 80% 单元测试
4. **配置驱动**：`zest.llm.adapters.*` 切换，符合运维习惯

### 5.2 实现成本估算

| SPI | 默认实现工时 | 替换实现工时 |
|-----|-------------|-------------|
| ModelGatewayAdapter | 3~5 人天 | +2 人天/实现 |
| ObservabilityAdapter | 2~3 人天 | +1~2 人天 |
| PromptRenderer | 1~2 人天 | +1 人天 |
| PolicyCacheAdapter | 2 人天 | +1 人天 |
| QuotaAdapter | 3 人天 | +2 人天 |
| AuditAdapter | 1 人天 | +1 人天 |
| ReportChannelAdapter | 2 人天 | +2 人天 |

**SPI 骨架 + LiteLLM + Noop 全套：约 2 人周**，在 v0.1 预算内。

### 5.3 潜在陷阱

| 陷阱 | 缓解 |
|------|------|
| SPI 接口过早抽象 | v0.1 仅实现必要方法；流式 chatStream v0.2 再加 |
| 配置项爆炸 | 统一 `zest.llm.adapters` 命名；文档 + Admin 健康页 |
| 多 Adapter 组合测试矩阵 | CI 只跑 default + noop 两套；替换实现由各 PR 自带测试 |

**结论：可插拔 SPI 可行，且是差异化竞争力，建议 v0.1 就引入接口骨架。**

## 6. Admin + Admin-UI 可行性

### 6.1 复用 ZestFlow Admin 模式

ZestFlow 已验证：

```text
zestflow-admin-ui (Vue) ──npm run build──► dist/
                                              │
                                              ▼
zestflow-admin/resources/static/  ◄── Maven copy-resources
```

ZestLLM 照搬该模式，**前端工程风险低**。

### 6.2 Admin 与 Runtime 合一 vs 拆分

| 方案 | 优点 | 缺点 | MVP 建议 |
|------|------|------|----------|
| **合一** | 部署简单、一套 Flyway、一套 JWT 体系 | 后期 CP 扩展需拆分 | ✅ v0.1 |
| **拆分** | CP 无状态水平扩展 | 两套部署、配置同步 | v0.3 |

**结论：** MVP 合一可行；代码层 **controller 分包 + 模块边界清晰**，便于后续物理拆分。

### 6.3 Admin-UI 页面工作量（MVP）

| 页面 | 工时 | 依赖 |
|------|------|------|
| 登录 + Layout | 2 人天 | 可复制 zestflow-admin-ui 骨架 |
| App / Task 列表 | 2 人天 | Admin API |
| Prompt 列表 + 编辑 | 3 人天 | Admin API |
| Execution 列表 + 详情 | 3 人天 | Admin API |
| Dashboard 占位 | 1 人天 | 可选 |

**合计约 1.5 人周**（1 前端），与后端 Admin API 并行。

### 6.4 阿里规范落地

参考 zestory-admin 已实践：

- `controller` / `service` / `repo` / `mapper` / `model` 分包
- `Result<T>` 统一响应
- Request 对象 + `@Valid`
- DO 后缀实体 + MyBatis-Plus

**无额外学习成本，可行。**

## 7. ZestFlow 元件 + MCP 开发可行性

### 7.1 MCP 发行状态

用户环境已具备（截图）：

| 文件 | 说明 |
|------|------|
| `zestflow-mcp-0.1.0-all.jar` | Fat JAR |
| `zestflow-mcp.jar` | 稳定入口 |
| `zestflow-dev-init.jar` | 项目初始化 |

安装路径：`~/.zestflow/tools/`（与 ZestFlow 官方 `install-mcp.ps1` 一致）

### 7.2 MCP 对 ZestLLM 的价值

| 能力 | 用途 |
|------|------|
| `scaffold_component` | 生成 Admin 治理 Handler 脚手架 |
| `list_components` | 防止编造 componentId |
| `validate_delivery` | AI 开发交付门禁 |
| `search_patterns` | 复用 paginated-list、admin-decision 等 Pattern |
| `init-dev` | 生成 `.zestflow/rules/architecture.md` + `.cursor/mcp.json` |

### 7.3 混合开发模式可行性

```text
80% 工作：Controller → Service → Repo（Admin-UI CRUD）  ← 传统阿里分层
20% 工作：@ZestComponent（复杂治理流程、与 ZestFlow 联调） ← MCP 辅助
100% 规范：MCP init-dev 注入禁止项 + validate_delivery 门禁
```

zestory 已在 `zestory-admin` 中同时使用 **Controller** 与 **@ZestComponent（AuthorHandler 等）**，证明混合模式可行。

### 7.4 MCP 限制（需团队知晓）

- MCP **不写盘**，源码需 IDE Apply
- MCP **不自动 publish** Prompt / 链到生产
- 需本地/远程 **Executor 或 Admin** 运行，`list_components` 才有效

**结论：MCP 元件式开发可行，且能显著降低 AI 辅助开发的失控风险。**

## 8. 性能与容量可行性

### 8.1 KPI 可达性分析

| KPI | 目标 | 可达性 | 关键措施 |
|-----|------|--------|----------|
| CP P95 < 50ms | 不含推理 | ✅ | prepare 只做 DB+缓存；不转发 token |
| traceId 100% | 全量 | ✅ | prepare 阶段生成 |
| fallback > 95% | 主模型失败 | ✅ | LiteLLM 原生 fallback |
| 业务接入 ≤ 0.5 天 | 人天 | ✅ | Starter 注解 + Demo |

### 8.2 瓶颈与对策

| 瓶颈 | v0.1 | v0.2+ |
|------|------|-------|
| CP 转发全部 token | invoke 合一可接受（MVP） | 必须三段式 |
| Execution 写入 | 同步 JDBC | Kafka 异步 report |
| Admin 与 CP 争抢 DB | 单库可接受 | 读写分离 / CP 拆分 |

### 8.3 容量粗算

假设：100 业务实例，每实例 10 QPS AI 调用

- **prepare/report**：2000 QPS → CP 2~4 实例（无推理，50ms 内）
- **execute**：1000 QPS → LiteLLM 3~5 实例（取决于模型供应商限额）
- **MySQL**：Execution 日增百万行 → 分区表 + 归档（v0.3）

**结论：架构上可支撑中等规模；超大规模需 v0.3 拆分 CP + 异步 report。**

### 12.3 验收用例

| # | 用例 | 步骤 | 预期 |

## 13. 决策建议

### 13.1 推荐方案（方案 A · 默认）

```text
ZestLLM = Admin（合一部署）+ SPI 可插拔 + LiteLLM 默认网关 + Langfuse v0.2 接入
开发 = Spring Boot + MyBatis-Plus + 阿里分层 + zestflow-mcp 元件规范
前端 = Vue3 Admin-UI（复用 zestflow-admin-ui 模式）
```

**理由：** 团队栈一致、开源底座成熟、SPI 保证后续可替换、MCP 降低 AI 开发风险。

### 13.2 备选方案

| 方案 | 适用 | 放弃原因 |
|------|------|----------|
| B：纯 LiteLLM，无 Admin | 个人项目 | 无企业治理 |
| C：采购 Portkey SaaS | 快速上线 | 不可控、难与 @ZestLLM 集成 |
| D：Spring AI 为中心 | Java 重度 | 缺 Control Plane 与 Prompt 运营 |

### 13.3 立即行动项（Next Steps）

1. **PoC 3~5 天**：LiteLLM + invoke + Starter 单测
2. **工程初始化**：`zestflow-dev-init --init-dev` + Maven 多模块骨架
3. **MCP 配置**：`.cursor/mcp.json` 指向 `zest-llm-admin` appCode
4. **docker-compose**：PG + Valkey + LiteLLM 最小集
5. **并行开发**：后端 Admin API + 前端 Prompt/Execution 页
